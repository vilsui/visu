Size and Complexity of Some DSLs:
- Konfiguraatiotyökalujen omat kielet eli DSL kasvavat yleensä erittäin laajoiksi ja vaikeaselkoisiksi.
- Salt-työkalu sisältää yli 510 funktiota joiden ohjekirja on laajempi kuin tyypillinen väitöskirja
- Salt käyttää Jinja2-malleja YAML-koodin luomiseen. Tämä lisää monimutkaisuutta kerroksen ylläpitäjän näkökulmasta
- Puppet sisältää yli sata (112) perusfunktiota mutta se käyttää ohjelmoinnissa omia perinteisiä kielistä poikkeavia käsitteitä funktioiden sijasta

Use of DSL Functions in Case Configuration (112-115. Mitä oikeasti käytetään.):

-Vaikka työkaluissa on useita ja jopa satoja ominaisuuksia, tuotannossa esim. Mozilla todellisuudessa käytössä vain kourallinen funktioita kattaa lähes 90% kaikesta käytöstä. 

- Package-file-Service on hallitseva peruskaava jolla suurin osa työstä on ohjelmien asentamista, asetusten muokkaamista ja palveluniden kynnistämistä.

- Exec-komentoa käytetään usein silloin, kun valmiit funktiot eivät ole riittäviä mmutta vaatii käyttäjältä huomiota idempotenssiin.
  
- Valintarakenteet if- sekä case on yleisimmin käytettyjä logiikan ohjaamiseen.

- Vähäisimmällä käytöllä on monia tärkeitä funktioita joita mm. käyttäjhien ja ryhmien hallintaan liittyviä funktioita
______________________________________________________________________________________________________
4.12.3.1 Dependencies Between Main Functions (115-117. Tärkeimmät rakennuspalikat.):

- Core functions: Kaikki hallinta perustuu loppujen lopuksi kahteen perustoimintoon jotka ovat file ja exec.

Tärkeimmät rakennuspalikat:
- Package-file-service: Standardi demonien pystyttämiseen.
- user & group: Käyttäjähallinta teknisiin tarpeisiin

- Idempotenssi on jörjestelmän keskeinen ominaisuus jossa muutoksia tehdään vain jos järjestelmä ei ole jo tavoitettavissa eli esim tiedosto luodaan vain jos se puuttuu.

- Riippuvuudert: Ylemmän tason toiminnot kuten package ja service kutsuvat taustalla järjestelmän omia työkaluja esim. apt-get. Exec toiminnon osalta idempotenssi on varmistettava erikseen.
  
- Abstraktio: Työkaluissa on satoja funktioita mutta suurin osa tarpeista mahdollista kattaa yhdistämällä peruspalikoita. Monimutkaisten uusien abstraktiokerrosten hyöty on yleensä kyseenalainen vaivaan nöähden.

B) Automaatti
Tämän jälkeen siirryin automatisoimaan PostgreSWQL-asennusta ansiblella. Siirryin ansible-kansioon: cd –/ansible jonka jälkeen loin kansion mkdir -p /roles/postgresql/tasks ja määritin nanolla roles /postpostgresql/tasks/main.yml jonka spekseisksi laitoin, sekä tallensin oheiset tiedot:

- name: Asenna postgresql
  apt:
    name: postgresql
    state: present
    update_cache: true
- name: Kaynnista postgresql
  service:
    name: postgresql
    state: started
    enabled: true

C) Asetus kohdassa siirryin muuttamaan PostgrSQL konffaustiedostoa ansiblella. Ensimmäisenä katsoin sijainnin komennolla sudo find /etc/postgresql -name "postgresql.conf" 

<img width="496" height="570" alt="image" src="https://github.com/user-attachments/assets/16f0c796-f01c-40ac-b713-f09dd2319338" />

Tiedoston löydettyäni kopioin sen ensin Ansible-rooliin mallitiedostoksi komennoin:
mkdir -p ~/ansible/roles/postgresql/files
sudo cp /etc/postgresql/17/main/postgresql.conf ~/ansible/roles/postgresql/files/
sudo chown visu:visu ~/ansible/roles/postgresql/files/postgresql.conf

D) Paikka remonttiin

Tässä tehtävässä varmistin että konffaus ansiblella on itsekorjautuva, eli eikun tuhoamaan osia ja selvitetään osaako ansible palauttaa kaiken ennalleen yhdellä komennolla. Tämä osottautui hyvin hankalaksi sillä ajoin terminaalissa komennon sudo rm -rf /etc/postgresql/17/main/ ja muut seuraavat komennot varmistaakseni että PostgreSQL on tuhottu. Tämä oli suuri virhe sillä se poisti sellaiset alikansiot joten jouduin tekemään paljon ylimääräistä luomalla conf.d kansion yms. muita tiedostoja
